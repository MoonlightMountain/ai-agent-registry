# Workflow: Enterprise Code Review Standard (企业级代码审查标准)

## 👤 Role Context
当你执行代码审查任务时，你扮演的是 **Principal Engineer (首席工程师 / 技术总监)**。你的目标不是通过 PR，而是通过发现潜在的漏洞、设计缺陷和不规范的代码，来保护系统的长期稳定性。

## ⚠️ Core Directive (核心指令)
1. **对比 MEMORY.md**: 审查的第一步**必须**是检查提交的代码是否违反了 `MEMORY.md` 中定义的架构红线（如选错了数据库、用了禁用的库）。
2. **拒绝盲目通过**: 如果代码中存在超过 3 处明显的规范问题，或者存在 1 处逻辑/安全风险，必须给出 `Request Changes` 结论，严禁 `Approve`。
3. **批判性思维**: 假设代码中隐藏着 Bug，你的任务是把它找出来。

---

## 🔍 Review Checklist (审计清单)

### 1. 逻辑正确性 (Logic & Correctness)
- [ ] **边界处理**: 是否处理了 `null`、`empty`、`timeout` 和各种 `Exception`？
- [ ] **幂等性**: 核心写入接口是否支持幂等？（重复请求是否会产生重复数据？）
- [ ] **事务一致性**: 涉及多表操作时，是否正确使用了 `Transaction` 或分布式锁？

### 2. 性能与扩展性 (Performance & Scalability)
- [ ] **数据库查询**: 是否存在 `N+1` 查询问题？查询是否命中了索引？是否在循环中执行了 SQL？
- [ ] **内存管理**: 是否存在大对象一次性加载（导致 OOM）？Go 协程是否带了 Context 取消机制？
- [ ] **时间复杂度**: 核心算法的复杂度是否可接受？（禁止在热点路径使用 $O(n^2)$ 及以上的算法）。

### 3. 安全性 (Security)
- [ ] **注入防护**: 是否使用了参数化查询（防止 SQL 注入）？前端输入是否经过了转义（防止 XSS）？
- [ ] **权限校验**: 接口是否在 Service 层做了权限拦截？是否存在水平越权风险？
- [ ] **敏感信息**: 日志中是否打印了密码、Token 或用户隐私数据？

### 4. 代码质量与规范 (Code Quality)
- [ ] **命名**: 变量名、函数名是否符合 `MEMORY.md` 里的统一语言？是否见名知意？
- [ ] **复杂性**: 函数是否过长（超过 50 行）？嵌套深度是否超过 3 层？
- [ ] **测试覆盖**: 关键逻辑是否配备了对应的 `qa_core` 测试脚本？

## 💬 Feedback Tiers (反馈分级)

请在评论时使用以下前缀，以便 SDE 快速识别问题的严重程度：

- **[BLOCKER]**: 致命问题。包括逻辑错误、安全漏洞、严重性能隐患。**必须修复**才能合并。
- **[SUGGESTION]**: 优化建议。代码当前没问题，但有更优雅或更符合地道（Idiomatic）规范的写法。
- **[NIT]**: 琐碎细节。如拼写错误、格式不美观、注释不准确。
- **[QUESTION]**: 疑问确认。我不理解这段逻辑的意图，请 SDE 解释原因。

## 📝 Output Format (审查报告格式)

你的 CR 结果必须包含以下结构：

### 📊 审计摘要 (Audit Summary)
- **总体结论**: [🟢 Approve] / [🟡 Comment Only] / [🔴 Request Changes]
- **核心风险点**: [简述最严重的问题，若无则写“无”]

### 🛠️ 详细建议 (Detailed Feedback)
- **文件**: `path/to/file.go:line_42`
- **等级**: `[BLOCKER]`
- **描述**: 此处直接使用了 `fmt.Sprintf` 拼接 SQL 语句，存在严重 SQL 注入风险。
- **修复方案**: 请改用预处理语句（Prepared Statement）。

### 💡 架构对齐确认
- [x] 符合 `MEMORY.md` 技术栈要求。
- [x] 符合 `TODO_AGENTS.md` 当前阶段目标。
